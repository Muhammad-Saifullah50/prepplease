# Implementation Plan: Foundation Services

**Branch**: `004-foundation-services` | **Date**: 2026-07-21 | **Spec**: `specs/004-foundation-services/spec.md`
**Input**: Feature specification for Course Core + Ingestion Pipeline FastAPI services — user auth via Clerk, course CRUD, file upload & ingestion, dashboard analytics.

## Summary

Build two FastAPI microservices (course-core on port 8001, ingestion-pipeline on port 8002) on top of the existing scaffold. The services expose CRUD endpoints for users and courses, file upload to S3 with async ingestion triggering, and dashboard/analytics views. Authentication uses Clerk's official `clerk-backend-api` Python SDK for JWT verification and webhook-based user sync. Background processing runs lightweight in-process tasks (no external queue). All endpoints follow the repository pattern, with pydantic v2 validation at every boundary.

## Technical Context

**Language/Version**: Python 3.12 (pinned via `.python-version`)

**Primary Dependencies**:
- `fastapi>=0.115`, `uvicorn[standard]`, `pydantic v2`, `pydantic-settings`
- `clerk-backend-api` (official Clerk Python SDK for JWT verification + backend API)
- `exambrain-shared` (workspace dep: db, config, S3Adapter, LLMClient, logging, errors)
- `exambrain-agents` (workspace dep: `ingest_course_file` pipeline, repositories)
- `aioboto3` (async S3 uploads — already in shared)
- `python-multipart` (file upload handling)
- `structlog` (JSON logging — already in shared)
- `prometheus-client` (metrics — already scaffolded)

**Storage**:
- PostgreSQL 17 + pgvector (3 databases: `course_core`, `ingestion`, `exam_sim`)
- AWS S3 (file storage via existing S3Adapter in shared)
- Redis 7 (future: event bus, rate limiting — provisioned but unused in this feature)

**Testing**: pytest + pytest-asyncio + httpx AsyncClient + ASGITransport. Clerk auth overridden via FastAPI `dependency_overrides` for unit tests.

**Target Platform**: Linux server (OCI Ampere A1 / K3s containers)

**Project Type**: Web (backend-only; frontend separate in later phase)

**Performance Goals**:
- Dashboard summary: p95 < 2s for users with 10 courses / 100 results (FR-022)
- Status polling: p95 < 500ms regardless of processing state (FR-023)
- Duplicate detection: < 1s (SC-003)

**Constraints**:
- Zero-cost infra: Oracle Always Free (4 OCPUs, 24GB RAM) + AWS Free Tier
- Total RAM footprint ~2.4GB across all services
- Async-first everywhere: no blocking I/O in request paths
- All DB access through repository layer (no raw SQL in handlers)
- Auth required by default on all endpoints (explicit public whitelist)

**Scale/Scope**: MVP — single-user-per-course model, no multi-tenant orgs. Target < 1000 active users in first release.
